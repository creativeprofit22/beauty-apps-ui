# Wiring Report: Spa UI Moodboard

Date: 2026-02-27
Scope: Full project (all 7 capabilities)

## Architecture Map

```
SKIN env var
    │
    ▼
next.config.ts ── webpack/turbopack alias ──▶ @/skin → skins/{SKIN}/
    │
    ▼
layout.tsx
  ├── import @/skin/skin.css ──▶ CSS token overrides (per-skin colors, radii, shadows, fonts)
  ├── import @/styles/globals.css
  │     ├── tokens.css ──▶ @theme inline (Tailwind bridge) + :root/:dark defaults
  │     ├── base.css ──▶ Reset, body, scrollbar, focus
  │     ├── effects.css ──▶ Keyframes, reduced-motion
  │     └── fonts.css ──▶ Font feature settings, gradient text
  ├── data-skin={SKIN} on <html>
  └── inline <script> ──▶ .dark class toggle (pre-hydration)

skins/_base/structure.css ──▶ Spacing, z-index, layout dims (NEVER IMPORTED)

Components (src/components/**)
  ├── CSS tokens via Tailwind classes (bg-surface-base, text-primary, etc.)
  ├── CSS tokens via var() in inline styles
  └── JS config via @/skin/config (skinConfig object)
```

## Capabilities Traced

---

### Capability 1: Skin CSS Token Propagation

- **Entry:** `SKIN` env var (default: `"spa"`)
- **Layers:** `next.config.ts` alias → `layout.tsx` import → skin.css override → Tailwind/var() consumption
- **Destination:** Component rendering via CSS custom properties

**Status: Mostly wired. 2 gaps found.**

| # | Where | What's Lost | Impact | Severity |
|---|-------|------------|--------|----------|
| 1 | `backgrounds/page.tsx:105,204`, `page.tsx:149` | Dark skin overrides don't apply to Cards with manual `class="dark"` | Non-spa skins in dark preview cards render with spa dark tokens, not their own | Moderate |
| 2 | `stat-card.tsx:51` | `--stat-glow` token referenced but never defined anywhere | Text shadow always resolves to `none` fallback | Minor |

**Details:**

**Gap 1 — Dark class on Cards vs data-skin selector:**
Skin CSS files use `.dark[data-skin="X"]` selectors, which only match `<html>`. When showcase pages apply `class="dark"` directly to a `<Card>` for preview purposes, that Card gets the generic `.dark` overrides from `tokens.css` (spa colors) but NOT the skin-specific dark tokens from `skin.css`. This affects barber, nail-salon, and tattoo dark preview cards — they render with spa-toned dark colors instead of their own palette.

**Gap 2 — Undefined `--stat-glow`:**
`stat-card.tsx:51` references `var(--stat-glow, none)`. This token doesn't exist in any CSS file. The `none` fallback means it's silently inert. Likely intended to use `--glow-primary` or similar.

### Capability 2: Skin Config (JS) Propagation

- **Entry:** `@/skin/config` import (aliased per skin)
- **Layers:** skin config.ts → page import → component props
- **Destination:** Confetti colors, scratch foil, card watermark

**Status: Core chain works. Dead exports + hardcoded defaults.**

| # | Where | What's Lost | Impact | Severity |
|---|-------|------------|--------|----------|
| 1 | All 4 `config.ts` files | `skinConfig.name` and `skinConfig.label` never consumed | Dead exports — no runtime cost but unfulfilled contract | Minor |
| 2 | `confetti.tsx:29` | Default `colors` prop hardcodes spa hex values | If `<Confetti>` rendered without explicit colors prop, shows spa palette on any skin | Minor |
| 3 | `useScratch.ts:39-44` | Default `foilGradient` hardcodes spa oklch values | Same pattern — defaults to spa foil if prop omitted | Minor |
| 4 | `loyalty-card.tsx:14-22` | `tierGradients` hardcoded oklch (spa-neutral) | Tier card gradients don't adapt to skin's `--tier-*` tokens | Moderate |
| 5 | `tier-upgrade.tsx:26-34` | `tierGradients` hardcoded oklch (spa-neutral) | Tier upgrade ink-bleed gradients don't adapt per skin | Moderate |

**Details:**

Gaps 2-3 are currently safe — callers always pass explicit props from `skinConfig`. But the hardcoded defaults are spa-specific, meaning any new usage of these components without explicit props will silently render wrong colors.

Gaps 4-5 are actual visual inconsistencies: tier gradients in `LoyaltyCard` and `TierUpgrade` use hardcoded oklch values that match the spa palette. The CSS tier tokens (`--tier-bronze`, `--tier-gold`, etc.) vary per skin, but these JS gradient definitions don't follow them.

### Capability 3: Dark Mode / Theme Wiring

- **Entry:** Inline script in `layout.tsx` (localStorage + system preference)
- **Layers:** `.dark` class on `<html>` → `useTheme` hook → CSS `.dark` overrides
- **Destination:** Token values flip for dark palette

**Status: Core wiring correct. 3 critical token gaps in skin dark blocks.**

| # | Where | What's Lost | Impact | Severity |
|---|-------|------------|--------|----------|
| 1 | `barber/skin.css` dark block (lines 99-175) | 6 radii tokens missing | Barber dark mode: sharp corners (2-12px) revert to spa soft corners (4-28px) | **Critical** |
| 2 | `nail-salon/skin.css` dark block (lines 107-195) | 6 radii tokens missing | Nail-salon dark mode: bubbly corners (8-36px) shrink to spa defaults (4-28px) | **Critical** |
| 3 | `tattoo/skin.css` dark block (lines 102-185) | 6 radii tokens missing | Tattoo dark mode: angular corners (1-6px) inflate to spa soft corners (4-28px) | **Critical** |
| 4 | `barber/skin.css` dark block | `--font-family-display` missing | Barber dark: Oswald → Space Grotesk | **Critical** |
| 5 | `nail-salon/skin.css` dark block | `--font-family-display` missing | Nail-salon dark: Quicksand → Space Grotesk | **Critical** |
| 6 | `tattoo/skin.css` dark block | `--font-family-display` missing | Tattoo dark: Bebas Neue → Space Grotesk | **Critical** |
| 7 | `barber/skin.css` dark block | `--glass-opacity` + `--gradient-intensity` missing | Undefined in dark barber (no fallback exists anywhere) | Moderate |

**Details:**

All three non-spa skins define radii and font-family-display in their light-mode blocks but omit them from their dark-mode blocks. CSS custom properties don't inherit between selector blocks — so dark mode falls back to `:root` defaults from `tokens.css`, which are spa values. The skin's geometric and typographic identity completely disappears in dark mode.

### Capability 5: Treatment Flag Consumption

- **Entry:** `--glass-opacity` and `--gradient-intensity` defined in all 4 skin.css files
- **Layers:** Should flow via `var()` to components
- **Destination:** Glass effects, gradient intensity modulation

**Status: 100% dead. Defined in 14 places, consumed in ZERO.**

| # | Where | What's Lost | Impact | Severity |
|---|-------|------------|--------|----------|
| 1 | All skin.css files (14 declarations) | Neither `var(--glass-opacity)` nor `var(--gradient-intensity)` appear anywhere in src/ | Skin personality flags (spa=glass, barber=matte, tattoo=flat) are completely inert | Moderate |

**Details:**

These tokens encode the design contract — spa should have glassmorphism and rich gradients, barber/tattoo should be matte and flat. But no component reads them. Components that logically should (Card, backgrounds page gradient panels, LoyaltyCard foil sheen, TierUpgrade gradients) all hardcode their own opacity/gradient values. The skin differentiation these flags promise is entirely undelivered.

Not in `@theme inline` either, so Tailwind utilities can't reference them.

### Capability 6: Font Override Loading

- **Entry:** Skin.css sets `--font-family-display` to skin-specific font
- **Layers:** CSS var → `font-family` property → browser font loading
- **Destination:** Display headings render in skin-appropriate typeface

**Status: BROKEN for all non-spa skins. Font packages not installed.**

| # | Where | What's Lost | Impact | Severity |
|---|-------|------------|--------|----------|
| 1 | `package.json` | `@fontsource-variable/oswald` not installed | Barber display font silently falls to Arial Narrow (Windows) or system-ui | **Critical** |
| 2 | `package.json` | `@fontsource-variable/quicksand` not installed | Nail-salon display font silently falls to system-ui | **Critical** |
| 3 | `package.json` | `@fontsource/bebas-neue` not installed | Tattoo display font falls to Impact (partial save) or system-ui | **Critical** |
| 4 | `layout.tsx` | No conditional font imports for non-spa skins | Even if packages were installed, they wouldn't be imported | **Critical** |

**Details:**

The CSS token wiring is correct — `--font-family-display` IS overridden in skin.css and IS consumed via `var()` in `fonts.css`. But the actual font files don't exist. The browser cascades through the font stack and lands on system fonts. Tattoo gets a partial save via Impact (system font), barber gets Arial Narrow on Windows, nail-salon gets nothing distinctive.

The fix requires: (1) installing 3 packages, (2) adding imports (either in layout.tsx conditionally, or in each skin.css via CSS @import).

## Systemic Patterns

### Pattern 1: Dark mode blocks are incomplete copies of light mode blocks
All three non-spa skins (barber, nail-salon, tattoo) define radii, font-family-display, and treatment flags in their light sections but omit them from their dark sections. This is a systematic copy-paste gap — the dark blocks were likely written by duplicating color overrides without carrying over geometry/typography/treatment tokens.

**Affected:** `barber/skin.css`, `nail-salon/skin.css`, `tattoo/skin.css` — 3 files, 20+ missing token declarations total.

### Pattern 2: Hardcoded spa-skin values in component defaults
Multiple components hardcode spa-palette values as defaults: `confetti.tsx` colors, `useScratch.ts` foil gradient, `loyalty-card.tsx` tier gradients, `tier-upgrade.tsx` tier gradients, `service-gallery.tsx` overlay. These work for the spa skin but don't adapt to other skins. The skinConfig system exists to solve this but only covers 3 of 5+ visual properties that vary per skin.

**Affected:** 5 component files with hardcoded spa-specific values.

### Pattern 3: Tokens defined but never consumed (dead infrastructure)
Two categories: (a) `--glass-opacity` and `--gradient-intensity` — 14 declarations across 4 skins, zero consumers; (b) `skinConfig.name` and `skinConfig.label` — exported from all 4 configs, imported nowhere. These represent design intent that was documented in token form but never wired to components.

### Pattern 4: CSS files defined but never imported
`skins/_base/structure.css` exists with 30+ well-structured tokens, consumed by 9+ components, but is never imported into the CSS cascade. The tokens resolve to undefined at runtime, causing layout dimensions, z-index stacking, and spacing to silently degrade.
